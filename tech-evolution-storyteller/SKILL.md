## 核心工作流

你的任務是將用戶提供的**技術主題**轉化為一份結構化、故事化的技術解釋文檔。

### Step 1: 提取與確認主題

接收用戶的輸入（格式：`/tech-evolution-storyteller "技術主題"`）。

**示例輸入**：
- "WiFi 無線傳輸如何工作"
- "HTTPS 如何保護數據"
- "GPU 為什麼能並行計算"
- "TCP 三次握手的原理"

### Step 2: 多層次 WebSearch 信息蒐集

進行**至少 3 輪搜索**，每輪搜索不同角度，確保覆蓋完整時間線：

**搜索策略**：
- **第 1 輪**：「[技術名] 歷史發展 起源」— 尋找早期發現、基礎原理、歷史時間點
- **第 2 輪**：「[技術名] 工作原理 詳解」— 尋找核心機制、技術細節、標準
- **第 3 輪**：「[技術名] 最新進展 應用場景」— 尋找現代實現、優化方向、實際應用

**信息擴展目標**：確保獲取至少 **6 個核心要點**，涵蓋：
- 2 個歷史背景（是誰發現的、什麼時候、為什麼需要）
- 2 個原理機制（基礎概念、關鍵步驟）
- 2 個現代應用（當代實現、優化技術）

### Step 3: 構建時間線架構

根據蒐集的信息，組織成以下時間軸：

```
【遠古時代】基礎科學發現
  ↓
【早期應用】第一代技術出現
  ↓
【演進階段】技術優化迭代
  ↓
【現代標準】主流技術方案
  ↓
【最新前沿】前沿研究與未來
```

### Step 4: 故事化講解與比喻構建

參考 `tech-personification-storytelling` 的風格，遵循以下原則：

**寫作風格**：
1. **從生活場景開頭** — 用日常場景引入（如「當你連上 WiFi 時」）
2. **逐步抽象化** — 從具體現象 → 物理原理 → 工程方案
3. **用類比和比喻** — 用日常熟悉的東西類比複雜概念
   - 水波比喻電磁波
   - 多車道比喻頻分復用
   - 多條耳朵比喻 MIMO
4. **嵌入歷史關鍵時刻** — 交插「某年某人發現」的故事點
5. **循序漸進的深度** — 先講基礎，再講優化，最後講極限
6. **回歸現實意義** — 最後說明這項技術如何影響日常生活

### Step 5: 生成 Markdown 文檔

輸出格式必須包含以下結構：

```markdown
# [技術名稱]：[簡潔主題句]

## 📍 一句話核心

[一句話概括這項技術最核心的貢獻]

## 🌍 時間線全景

[Mermaid 時間線圖表或表格，列出 6+ 核心要點的年份和事件]

## 🧩 故事化講解

### 第一幕：基礎概念
[從日常場景出發，引入基礎物理原理。用生活比喻。1000-1500 字]

### 第二幕：技術演進
[展示技術如何從簡單逐步優化複雜。列舉 2-3 個關鍵的演進步驟。800-1200 字]

### 第三幕：現代應用
[講述當代實現方案、標準化、最新優化。1000-1500 字]

## 📊 概念圖解

[使用 Mermaid 或 ASCII 圖表可視化核心概念]

## 🔗 引用與圖片

[列出重要來源、引用標準文檔連結、相關圖片連結]
- [標準/文獻]：URL
- [官方圖示]：描述 + URL（如 Wikipedia、官方技術文檔、論文）

## 💡 核心洞見

- **要點 1**：[技術創新的本質]
- **要點 2**：[物理極限與工程平衡]
- **要點 3**：[未來發展方向]

## 🎓 進階探索

[如果用戶想深入，還有哪些相關主題可以學習]
```

### Step 6: 圖表與視覺化要求

**必須包含**：
- **Mermaid 時間線或流程圖** — 展示技術從發現到現在的演進或工作原理
- **ASCII 圖解或文字圖表** — 用 ASCII 藝術或結構化文字視覺化核心概念
- **表格或對比圖** — 展示技術迭代、版本演進、性能對比

**圖片處理方式**（只用 Obsidian 本地引用）：

1. **搜尋並下載圖片** → 自動保存到 `/Users/bart/obsidian-vault/12.資源/12.3.圖片資產`
   - 優先從 Wikimedia Commons、Wikipedia、官方技術文檔搜尋
   - 支持格式：PNG、SVG、JPG
   - 建議搜尋關鍵字：`[技術名] [內容] diagram site:commons.wikimedia.org`

2. **在 Markdown 中用 Obsidian 本地引用語法**：
   ```markdown
   ![[图片文件名.png]]
   或
   ![[图片文件名.svg]]
   ```
   - 圖片自動從 `/Users/bart/obsidian-vault/12.資源/12.3.圖片資產` 中引用
   - Obsidian 會自動解析並嵌入顯示

3. **圖片命名規則**：
   - 用中文描述，清晰易讀（如 `TLS_1.2_握手.svg`, `GPU_SIMT_架構.png`）
   - 避免特殊符號和長名稱

4. **無法找到圖片時**：
   - 用 Mermaid 圖表或 ASCII 圖解替代（已有支持）
   - 不使用外部連結或圖片資源清單

### Step 7: 輸出內容精簡

最終 Markdown 文檔**不包含**以下項目：
- ❌ 「🔍 數據驗證」或「檢查清單」（內部自檢用，不放入最終文檔）
- ❌ 尾部的「核心要點檢查清單」

只輸出用戶看得到的內容：故事、圖表、引用、洞見。

## 使用示例

**用戶輸入**：
```
用 tech-evolution-storyteller 講解「USB 協議如何進化」
```

**Skill 應做**：
1. 搜索 USB 的歷史（Intel 1996 推出）、演進（USB 2.0 → 3.0 → 3.1 → 4.0）、最新進展
2. 梳理核心要點：為什麼需要 USB → 速率瓶頸 → 功率傳輸優化 → Type-C 統一
3. 寫成故事：從「你的手機充電線」開始 → USB 如何解決接口混亂 → 各代速率升級的物理原理 → 當代 USB4 的意義
4. 生成完整 Markdown，包含時間線圖表、概念圖、引用原始規格

## 特別提醒

- **信息來源**：優先從學術文獻、官方規格、知名技術媒體（如 IEEE、ACM、廠商白皮書）蒐集
- **比喻貼切**：比喻要能幫助理解，不要為了比喻而比喻。寧可詳細解釋也不要誤導
- **時間線準確**：核心歷史日期應該準確（可以標註「約」字）
- **圖表清晰**：Mermaid 圖表應該簡潔可讀，複雜的用分層設計
- **字數平衡**：每個 Marmaid 故事段落 800-1500 字，整體文檔 3500-5000 字為佳
